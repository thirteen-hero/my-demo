## React+Node大文件分片上传、断点续传、秒传(封装组件)
### 思路
一、分片上传
  1、前端分片切割: 使用File.slice()将文件按固定大小如1MB切割成多个Blob分片(本组件默认5MB,实验为较优状态)
  2、生成唯一标识: 通过SparkMd5库计算文件内容的哈希值,作为文件唯一id(用于校验文件是否已存在)
    注意：分片大小不同,计算出的md5值不同
  3、并发控制: 通过limitLoad控制单次请求数量(本组件默认单次请求数量为4)
二、秒传
  校验文件是否已上传: 每次上传文件时计算文件哈希值,先通过文件名称查询文件是否已存在,若已存在则通知秒传；若不存在则校验进行断点续传或分片上传
三、断点续传
  上传前先查询服务端,获取已上传分片列表,若未上传分片,则对全部分片进行上传；若上传了部分分片,则跳过上传部分,进行断点续传
三、分片合并
  1、服务端按序合并: 收到合并请求后,按分片索引顺序将临时分片合并为完整文件
  2、清理临时文件: 合并完成后删除临时分片,释放存储空间

### 知识点
一、spark-md5
  spark-md5是一个js库,用于快速计算文件或数据的md5值,支持浏览器环境,可用于文件完整性校验和分片计算。其主要功能包括append、calculate、end和init等方法,适用于大文件分片处理。由于md5是单向哈希,故无法解密。通过比较文件的md5值,可以判断文件是否一致。
  spark-md5根据md5算法将任意长度的明文,经过一系列操作,转换成一个128位(16字节)的密文。
  spark-md5实现了在浏览器中进行哈希计算,并且在上传文件时有效检测文件是否被篡改。
  md5算法的实现过程：
    1、填充原始消息,使其长度对512bits取余为448bits。
    2、将填充好的消息和64位代表原始消息长度的数拼接到一起,形成一个新的消息。新的消息被划分为若干个512bits的分组,每个分组进行额外运算得到128位的输出。
    3、对每个分组进行一系列转换操作,逐个更新状态(128位矢量),生成最终状态的输出。
    4、将第三步得到的128位的16进制数转换为32位的字符串,即md5的值。
  spark-md5计算出来的md5值可以用于比较文件是否一致。如果两个文件的md5值相同,则可以确定这两个文件完全一样。
二、FileReader(FileReader是异步操作)
  FileReader接口允许web应用程序异步读取存储在用户计算机上的文件(或原始数据缓冲区)的内容,使用File或Blob对象指定要读取的文件或数据。
  文件对象可以从用户使用<input type="file" />元素选择文件返回的FileList对象中读取,或者从拖放操作的DataTransfer对象中获取。FileReader只能访问用户明确选择的文件内容,它不能用于从用户的文件系统中按照路径名读取文件。
  FileReader.readAsArrayBuffer()
  开始读取指定的Blob或File中的内容,一旦完成,readyState属性变为done,并触发loadend事件。result属性中将包含一个表示文件数据的ArrayBuffer对象。
  load事件: 在成功读取文件时触发。使用addEventListener监听load事件或设置事件处理器属性onload。(此事件不可取消且不会冒泡)
  loadstart事件: 在文件读取操作开始时触发。使用addEventListener监听loadstart事件或设置事件处理器属性onloadstart。(此事件不可取消且不会冒泡)
  loadend事件: 在读取文件完成时触发。(无论成功与否)使用addEventListener监听loadend事件或设置事件处理器属性onloadend。(此事件不可取消且不会冒泡)
  error事件: 在由于错误(例如文件未找到或不可读)导致读取失败时触发。使用addEventListener监听error事件或设置事件处理器属性onerror。(此事件不可取消且不会冒泡)(此事件不可取消且不会冒泡)
三、ArrayBuffer
  ArrayBuffer用来表示通用的原始二进制数据缓冲区。它是一个字节数组。不能直接操作ArrayBuffer中的内容,而是要通过类型化数组对象或DataView对象来操作,他们会将缓冲区的数据表示为特定的格式,并通过这些格式来读取缓冲区的内容。
  ArrayBuffer()构造函数创建一个以字节为单位的给定长度的新ArrayBuffer。也可以从现有的数据(如Base64字符串或从本地文件(fileReader.readAsArrayBuffer))获取数组缓冲区。
  ArrayBuffer是一个可转移对象。
  ArrayBuffer可以在不同的执行上下文之间传输,就像web worker或service worker那样,使用结构化克隆算法。这可以通过在Worker.postMessage()或ServiceWorker.postMessage()的调用中传入ArrayBuffer对象作为可转移对象来完成。在纯js中,也可以使用transfer()或transferToFixedLength()方法来转移内存的所有权。
  当一个ArrayBuffer对象被传输时,它原来的副本会被分离(detached),这代表它不再可用。在任何时候,只有一个ArrayBuffer副本实际拥有底层内存。分离的缓冲区具有一下行为:
    1、byteLength变为0(在缓冲区和关联的类型化数组视图中)
    2、所有实例方法,比如resize()和slice(),会在调用时抛出TypeError。关联的类型化数组视图的方法也会抛出TypeError。
  可以通过其detached属性来检查ArrayBuffer是否已分离。
四、可转移对象
  可转移对象是拥有属于自己的资源的对象,这些资源可以从一个上下文转移到另一个,确保一次仅在一个上下文可用。传输后原始对象不再可用。它不再指向转移后的资源,并且任何读取或者写入该对象的尝试都将抛出异常。
  可转移对象通常用于共享资源,该资源一次仅能安全地暴露在一个js线程中。例如,ArrayBuffer是一个拥有内存块的可转移对象。当此类缓冲区在线程之间传输时,相关联的内存资源将从原始的缓冲区分离出来,并且附加到新线程创建出的缓冲区对象中。原始线程的缓冲区不再可用,因为它不再拥有属于自己的内存资源了。
  使用转移对象资源的机制取决于对象自身。例如,当ArrayBuffer在线程之间转移时,它指向的内存资源实际上以快速且高效的零拷贝操作在上下文之间移动。其他对象可以通过拷贝关联的资源,然后将它从旧的上下文中删除来转移它。
五、FileList
  FileList接口表示<input />元素的files属性返回的该类型的对象。这样可以访问使用<input type='file' />元素选择的文件列表。
  实例属性: length 表示列表中文件数量的只读值
  实例方法: item() 返回一个File对象,表示文件列表中指示索引处的文件
六、Blob
  Blob对象表示一个不可变、原始数据的类文件对象。它的格式可以按文本或二进制的格式进行读取,也可以转换成ReadableStream来用于数据操作。
  Blob表示的不一定是js原生格式的数据。File接口基于Blob, 继承了blob的功能并将其扩展以支持用户系统上的文件。
  要从其他非blob对象和数据构造一个Blob, 要使用Blob()构造函数。要创建一个blob数据的子集blob,要使用slice()方法。
  Blob.slice()返回一个新的Blob对象,其中包含调用它的blob的指定字节范围内的数据。
七、File(File接口继承了Blob接口的属性)
  File接口提供有关文件的信息,并允许网页中的js访问其内容。
  File对象通常从用户使用<input type='file' />元素选择文件返回的FileList对象中检索,或者从拖放操作返回的DataTransfer对象中检索。
  File对象是一种特定类型的Blob,并且可以在Blob可以使用的任何上下文中使用。
八、Web Worker
  Web Worker为web内容在后台线程中运行脚本提供了一种简单的方法。线程可以执行任务而不干扰用户界面。它们可以使用XMLHttpRequest或fetch执行I/O。一旦创建,一个worker可以将消息发送到创建它的js代码,通过将消息发布到该代码指定的事件处理器。
  一个worker是使用一个构造函数创建的一个对象运行一个命名的js文件————这个文件包含将在worker线程中运行的代码。worker运行在另一个全局上下文中,不同于当前的window。因此,在worker内通过window获取全局作用域(而不是self)将返回错误。
  在专用worker的情况下,DedicatedWorkerGloblScope对象代表了worker的上下文(专用worker是指专有worker仅在单一脚本中被使用;共享worker的上下文是SharedWorkerGlobalScope对象)。一个专用worker仅能被首次生成它的脚本使用,而共享worker可以同时被多个脚本使用。
  在worker内,不能直接操作DOM节点,也不能使用window对象的默认属性和方法。但是可以使用大量window对象之下的东西,包括webSocket、indexDB等数据存储机制。
  workers和主线程之间都通过postMessage方法发送各自的消息,使用onMessage事件处理函数来响应消息(消息被包含在message事件的data属性中)。这个过程中数据不是被共享,而是被复制。
  只要运行在同源的父页面中,worker可以依次生成新的worker;并且可以使用XMLHttpRequest进行网络I/O,但是XMLHttpRequest的responseXML和channel属性总会返回null。
  一、专用worker: 
    1、定义: 一个专用worker仅能被生成它的脚本所使用。
    2、生成一个专用worker: const myWorker = new Worker('worker.js');
    3、专用worker中消息的接收和发送: 
      在主线程中使用时,onMessage和postMessage必须挂在worker对象上,而在worker中使用时不用这样做。因为,在worker内部,worker是有效的全局作用域。
      当一个消息在worker和主线程之间传递时,它被复制或者转移了,而不是共享。
    4、终止worker: myWorker.terminate();
    5、处理错误: 当worker出现运行中错误时,它的onerror事件处理函数会被调用。
    生成subworker:
    subworker必须托管在同源的父页面内。subworker在解析URI时会相对于父worker的地址而不是自身页面的地址。这使得worker更容易记录它们之间的页面关系。
    worker能够访问一个全局函数importScripts()来引入脚本,该函数接收0个或者多个URI作为参数来引入资源。
    importScripts('foo.js', 'main.js');
    浏览器加载并运行每一个列出的脚本。每个脚本中的全局对象都能被worker使用。如果脚本无法加载,将抛出NETWORK_ERROR异常,接下来的代码也无法执行。而之前执行的代码（包括使用setTimeout()异步执行的代码）依然能够执行。importScripts()之后的函数声明依然会保留,因为它们始终会在其他代码之前运行。
    脚本的下载顺序不固定,但执行时会按照传入importScripts()中的文件名顺序进行。这个过程是同步完成的；直到所有脚本都下载并执行完毕,importScripts()才会返回。
  二、共享worker
    1、定义: 一个共享worker可以被多个脚本使用————即使这些脚本正在被不同的window、iframe或者worker访问。
    2、注意:
      如果共享worker可以被多个浏览上下文调用,所有这些浏览上下文必须属于同源。
      在fireFox中,共享worker不能被私有和非私有window对象的document所共享。
    3、生成一个共享worker: const myWorker = new SharedWorker('worker.js');
    4、与一个共享worker通信必须通过port对象————一个确切的打开的端口供脚本与worker通信。
    在传递消息之前,端口连接必须被显式的打开,打开方式是使用onmessage事件处理函数或者start()方法。只有一种情况下需要调用start()方法,那就是message事件被addEventListener()方法使用。
    在使用start()方法打开端口连接时,如果父级线程和worker线程需要双向通信,那么它们都需要使用该方法。
    5、共享worker中消息的接收和发送
      当一个端口连接被创建时,使用onconnect事件处理函数来执行代码。使用事件的ports属性来获取端口并存储在变量中。为端口添加一个onmessage事件处理函数用来做运算并回传结果给主线程。在worker线程中设置此onmessage处理函数也会隐式的打开与主线程的端口连接。
  三、worker中数据的接收和发送: 在主页面和worker之间传递的数据是通过拷贝,而不是共享来完成的。传递给worker的对象需要经过序列化,接下来在另一端还需要反序列化。页面与worker不会共享同一个实例,最终的结果就是在每次通信结束时生成了数据的一个副本。大部分浏览器使用结构化克隆来实现该特性。
  结构化克隆算法可以接收JSON数据以及一些JSON不能表示的数据————比如循环引用。
  worker在后台执行运算,一个优势在于能够执行处理器密集型的运算而不会阻塞UI线程。
  在一个worker中最主要的不能做的事情就是直接影响父页面。包括操作父页面的节点以及使用页面中的对象。只可以间接地实现,通过DedicatedWorkGlobalScope.postMessage回传消息给主脚本,然后从主脚本那里执行操作或变化。
九、结构化克隆
  用于复制复杂js对象的算法。通过来自worker的postMessage()或使用IndexDB存储对象时在内部使用。它通过递归输入对象来构建克隆,同时保持先前访问过的引用的映射,以避免无限遍历循环。
十、浏览器消息循环
  js代码执行和浏览器渲染都发生在浏览器渲染主线程上,而且他们是单线程的,也就是说js代码运行一定会阻塞浏览器渲染。为了解决这个问题,渲染主线程永久轮询执行任务；单独开辟一个消息队列空间存放各个线程加入的任务。消息队列的无限轮询就叫消息循环,消息循环就发生在渲染主线程里。
  任务本身没有优先级,渲染主线程依次从队头拿取任务执行。有优先级的是消息队列。
  消息队列的优先级为: 微任务队列 > 交互队列 > 延时队列
### 优化
针对小size文件 影响不大

针对大文件的优化：
问题: 分片哈希计算速度过慢,等待时间过长
原因: 计算文件分片hash是一个cpu密集型任务,直接在主线程中计算hash会阻塞页面响应
优化: 放到web worker中计算hash
原因: promise本质上是微任务队列,单线程串行计算。但web worker本质上是并行计算,可以充分利用cpu多核心。
为什么不将分片流程也放入web worker?
答: 
  1、切片过程不会消耗太多时间,他只是创建引用,不复制数据
  2、真正的阻塞点在读取时,但I/O操作是异步的,Blob转ArrayBuffer的过程基于promise,这并不会阻塞主线程
  3、如果将这个过程放入web worker,由于File和Blob不是可转移对象,此时会导致主线程与worker进行通信时进行结构化克隆,由此会产生额外的cpu性能消耗和内存消耗
  4、并未一次性创建大量异步任务,而是分组创建,控制了内存占用情况,也提高了效率

问题: 大文件合并后文件损坏
报错: The value of "length" is out of range. It must be >= 0 && <= 2147483647. Received 3183801400
原因: concat-files底层调用fs.writeFile来进行文件写入,写入文件大小阈值为0-2G,当写入文件过大时,就会报错
解决方案: 通过递归的方式对文件分片进行流式写入,避免一次性将大量数据加载到内存中。

问题: 文件上传采用Promise.all的方式,当文件过大时会同时开启多个http请求,有可能导致后续请求阻塞,甚至浏览器卡顿或崩溃
解决方案: 采用限制最大请求数量的方案发送请求, Promise.all和Promise.race结合使用(控制并发数量,采用promise.race对一组请求进行并发,一个返回就讲一个新的加入,保证http请求以限制数量进行并发请求)
注意: 当上传文件较大时(比如3.18G),要注意分片大小和最大请求数量大小,当分片过小时会报错(如1*1024*1024),建议分片大小为5*1024*1024

问题: 请求报错
解决方案: 重试一次,仍失败的话就不管了。其他接口照常请求,但ui提示文件上传失败,等重新上传时,只上传之前上传失败的部分。

问题: 进度条问题
已解决

问题: 部分视频文件无法播放视频 只能播放音频
经查证: 文件问题 非是代码问题

图片预览
已实现

问题: 采用dateUrl和objectUrl的方案嵌入将web worker代码嵌入js字符串中 无法工作
解决方案: import 引入的spark-md5包应采用cdn引入方案 即可

问题: 上传完毕后卡顿
原因: 文件读取问题 虽然是异步读取 但是会被判断为长任务 造成卡顿
解决方案: 采用URL.createObjectUrl()的方式读取

当时我们增长业务线提出了通过后台配置录播视频展示在主站资源位进行引流，我这边的工作内容就是支持后台上传视频资源。但是由于业务方无法确定视频资源的大小，给定范围在几十MB到几个GB之间。在过往的开发经验中我了解采用传统的表单上传的方案用户体验极差，也有可能在视频资源即将上传成功时由于网络波动需要重新上传。所以当时我的开发目标就是封装一个高可靠高性能的上传组件，基于分片上传的思想，支持视频、音频、图片等资源上传，也支持上传进度展示、资源预览、断点续传、文件秒传功能。
最初我的技术方案是，对资源文件进行分片并基于Promise串行计算每个分片的md5，同时采用混合调度模式进行并发控制来上传分片，平衡带宽利用率和浏览器压力，并配合后端的文件指纹机制实现了断点续传和秒传功能。
但是实现后发现，针对较小的资源比如图片音频体验尚可，但是对于较大的视频资源，比如，3G的视频计算md5到上传完毕耗时达到41s。而且一个分片上传失败后就注定本次文件上传失败，因为由于缺失分片，后端无法对分片进行合并。而且请求超时也会阻塞网络。
因此对于这个组件又进行了技术方案的升级。针对分片md5计算速度慢的问题，决定充分利用cpu多核心的特点，基于web worker并行计算每个分片的md5值。我设计了一个基于二维数组的批次调度策略，外层控制轮数，内层控制并发，同时为了减小频繁创建和销毁web worker的开销，我写了一个WorkerPool，对并发的worker进行复用。同时通过发布订阅的方式监听计算进度和控制轮次进展。针对文件上传失败导致分片合并失败的问题，我针对每个分片上传都做了失败重试，并且为了防止多请求同时重试再同时失败的问题，我做了基于指数退避的延时重试。针对请求超时的问题，我利用AbortController和setTimeout实现了请求超时取消，防止无效请求占用网络资源。同时对所有请求失败的情况做了统一兜底处理，保证了上传过程的健壮性。
做了这些调整后发现3G的视频资源从计算md5到上传完毕耗时仅需16s，速度提高了60%。但同时发现了一个问题，上传成功后页面卡顿，我通过chrome performance面板发现读取文件资源进行预览是个长任务，采用objectUrl的方案代替了FileReader.readAsUrl，因为后者会将视频资源进行base64编码，导致文件体积膨胀30%，引发页面卡顿。
最后，在工程化落地方面，为了规避Webpack打包路径配置可能导致的线上404风险，我没有单独创建Worker JS文件，而是将Worker代码以字符串形式内嵌，配合URL.createObjectURL动态注入。
最终，这个组件将大文件上传的成功率提升到了99%以上，不仅解决了超时问题，还作为通用基建复用到团队其他图片、音频上传场景中，统一了团队的上传标准。

背景：所处增长业务线提出配置录屏视频资源展示在主站主页资源位的需求，由于业务方无法确定视频资源大小，给定范围在几十MB到几个GB之间，资源配置后台需提供视频上传和配置相关信息的功能。
初步技术方案：
封装一个上传组件，支持图片、视频、音频资源上传。采用分片上传方案，自定义分片大小和并发请求数量，支持预览、展示进度条、断点续传和秒传。
实现：
  1、采用useReducer初始化、更新 计算cdn进度条和上传文件分片进度条
  2、并发通过创建一个promise set 配合Promise.all和Promise.race
  3、FileReader.readAsUrl()实现资源预览
  4、基于promise串行计算md5
问题： 
  1、分片哈希计算速度过慢，等待时间过长
  2、一个文件上传失败后，造成合并失败
  3、请求超时造成的资源占用和网络阻塞
解决方案：
  1、采用web-worker，并行计算分片哈希
     实现：
        1、文件切片在主线程进行 将切片放入二维数组
        2、采用二维数组的方案 遍历外层数组 内层数组并发 等待内层并发全部完成再处理下一轮
        3、根据内层数组的数量创建web worker实例 采用创建workerPool类的方案 可以达到复用之前的web worker实例 不必重复创建
        4、结合发布订阅处理任务进出
        5、全部分片计算完毕后 关闭web worker
        6、对全部的分片哈希进行二次哈希 标识文件唯一性
     成果：计算速度提高60%
  2、支持失败重试 延迟重试 指数退避
     实现：在catch中递归调用失败请求 通过延迟重试和指数退避减少请求失败和弱网情况下的请求问题
  3、超时取消
     实现：通过setTimeout计时，通过AbortController()signal标识axios，超时取消
  4、对请求失败情况进行了统一处理
问题：文件上传成功后页面卡顿
原因：预览视频资源采用FileReader.readAsDataUrl()读取，会转码成base64，造成文件体积膨胀
解决方案：采用URL.createObjectUrl()创建url和blob的映射
将web worker代码以js字符串的形式写入代码中 没有单独创建js文件
